# 프로젝트 요구사항 및 설계 표준 가이드라인 (Full Version)

## 1. 요구사항 분석 및 정의 (Analysis & Definition)
- 모든 기능 요구사항은 '누가, 왜, 무엇을' 하는지 명확히 기술한다.
- 비즈니스 목적(Goal)이 기술되지 않은 기능은 포함하지 않는다.
- 사용자의 페르소나를 최소 1개 이상 정의하고 그 관점에서 서술한다.
- 모든 요구사항은 고유 ID(예: REQ-001)를 부여한다.
- 요구사항 간의 우선순위(Must, Should, Could, Won't)를 지정한다.
- 주관적인 형용사(적당히, 빠르게 등) 사용을 금지하고 정량적 수치를 사용한다.
- 해당 서비스가 해결하려는 핵심 문제(Problem Statement)를 최상단에 명시한다.
- 기존 서비스와의 차별점(Unique Selling Point)을 한 문장으로 정의한다.
- 서비스의 범위(In-Scope)와 범위 밖(Out-of-Scope)을 명확히 구분한다.
- 타겟 시장과 주 사용 층의 기술 숙련도를 고려하여 용어를 선택한다.
- [추가] 모든 기능은 법적 규제(개인정보보호법 등)에 저촉되는지 사전 검토한다.
- [추가] 신규 기능 추가 시 기존 기능과의 하위 호환성(Backwards Compatibility)을 검토한다.

## 2. 논리적 구조 및 프로세스 (Logic & Process)
- 핵심 사용자 여정(Happy Path)을 단계별로 기술한다.
- 예외 케이스(Edge Case)에 대한 처리 로직을 반드시 포함한다.
- 각 기능의 시작 조건(Pre-condition)을 명시한다.
- 각 기능의 종료 조건(Post-condition)을 명시한다.
- 데이터의 생성(C), 조회(R), 수정(U), 삭제(D) 생명주기를 정의한다.
- 사용자의 입력 오류 발생 시 복구 시나리오를 정의한다.
- 시스템 간 데이터 연동이 필요한 경우 인터페이스 방식을 기술한다.
- 비동기 처리(Background Process)가 필요한 부분을 식별한다.
- 중복 데이터 입력을 방지하기 위한 로직을 설계한다.
- 각 프로세스의 타임아웃(Timeout) 기준을 설정한다.
- [강화] 다단계 프로세스(Wizard 패턴)의 경우, 중간 저장(Draft) 시점을 정의한다.
- [강화] 조건별 분기 로직이 3개 이상일 경우 의사결정 테이블(Decision Table)을 포함한다.

## 3. 데이터 및 정보 설계 (Data & Information)
- 서비스에서 사용하는 핵심 엔티티(Entity)를 정의한다.
- 각 엔티티 간의 관계(1:1, 1:N, N:M)를 명시한다.
- 필수 입력 값과 선택 입력 값을 구분한다.
- 데이터 필드의 타입(String, Number, Date 등)을 지정한다.
- 데이터의 최대 길이(Max Length) 및 유효성 검사 규칙을 정의한다.
- 날짜와 시간은 표준 타임존(UTC) 처리 방식을 준수한다.
- 통화나 단위 변환이 필요한 경우 기준 단위를 명시한다.
- 개인정보(PII)에 해당하는 데이터 항목을 식별한다.
- 데이터 삭제 시 물리적 삭제(Hard)와 논리적 삭제(Soft)를 구분한다.
- 검색 기능이 있다면 검색 필터와 정렬 기준을 정의한다.
- [추가] AI 모델 학습에 사용되는 데이터와 휘발성 데이터를 명확히 구분한다.
- [추가] 데이터 마스킹(Masking)이 필요한 민감 정보 리스트를 정의한다.

## 4. 사용자 경험 및 인터페이스 논리 (UX Logic)
- 모든 상태(Loading, Success, Empty, Error)에 대한 대응책을 마련한다.
- 페이지 이동(Navigation)의 흐름도를 텍스트나 다이어그램으로 설명한다.
- 뒤로 가기(Back) 버튼 클릭 시의 동작을 정의한다.
- 데이터가 없을 때 표시할 'Empty State' 문구를 정의한다.
- 로딩 시간이 길어질 경우의 피드백 방식(Skeleton 등)을 제안한다.
- 모바일, 태블릿, PC 등 멀티 디바이스 대응 정책을 수립한다.
- 접근성(Accessibility)을 고려한 기본 준수 사항을 명시한다.
- 알림(Notification) 발생 조건과 전달 채널을 정의한다.
- 파괴적 액션(삭제 등)에는 사용자 확인(Confirmation) 단계를 넣는다.
- 툴팁이나 가이드가 필요한 복잡한 기능을 식별한다.
- [강화] 모든 인터랙션 피드백은 0.2초(200ms) 이내에 시각적으로 반응해야 한다.
- [강화] 폼(Form) 입력 시 자동 완성(Auto-complete) 가능 여부를 명시한다.

## 5. 비기능적 요구사항 (Non-Functional)
- 동시 접속자 수 예상치 및 부하 처리 목표를 기술한다.
- 시스템 응답 속도(Latency)의 허용 범위를 정의한다.
- 서비스 가용성(Uptime) 목표치를 설정한다.
- 보안 요구사항(인증, 인가, 암호화)을 기술한다.
- 로그 기록 정책(무엇을, 얼마나 보관할지)을 수립한다.
- 장애 발생 시 알림 수신 대상과 절차를 정의한다.
- 백업 및 복구 전략(RTO, RPO)을 명시한다.
