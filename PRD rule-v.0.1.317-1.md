1. 제품 전략 및 가치 정의 (Strategy & Value)
모든 프로젝트는 해결하고자 하는 **핵심 문제(Problem Statement)**를 최상단에 정의한다.

서비스의 존재 이유를 증명하는 **독보적 가치 제안(USP)**을 한 문장으로 명시한다.

비즈니스 목적(Goal)과 연결되지 않는 기능은 요구사항에서 배제한다.

기능의 성공 여부를 판단할 **정량적 KPI(성공 지표)**를 최소 2개 이상 설정한다.

프로젝트의 업무 범위(In-Scope)를 명확히 하여 일정 리스크를 방지한다.

업무 범위 외 사항(Out-of-Scope)을 명시하여 기획-개발 간 오해를 차단한다.

서비스의 브랜드 톤앤매너(Brand Voice)를 정의하여 UI 문구의 일관성을 유지한다.

시장 내 경쟁 서비스 대비 우위 요소와 열위 요소를 분석하여 정책에 반영한다.

서비스의 단기 출시 목표와 장기적 확장 방향성(Roadmap)을 요약한다.

수익 모델(Monetization)이 있는 경우, 과금 지점과 단가 정책을 명시한다.

주요 검색 키워드 리스트를 정의하여 서비스 발견성을 확보한다.

페이지별 메타 정보(Title, Description) 및 OG Tag 노출 정책을 수립한다.

서비스 슬로건과 핵심 키워드를 정의하여 사용자에게 전달할 메시지를 통일한다.

신규 기능 도입 시 기존 사용자 환경에 미치는 영향도를 분석한다.

타겟 시장의 법적 규제 및 산업 특수성을 전략 단계에서 사전 검토한다.

2. 타겟 사용자 및 접근성 (Target & Accessibility)
서비스의 주 사용자를 대변하는 **페르소나(Persona)**를 구체화한다.

사용자의 기술 숙련도를 고려하여 전문 용어와 일반 용어 사용 비중을 결정한다.

저시력자를 위한 폰트 크기 가변성 및 고대비 모드 지원 정책을 수립한다.

스크린 리더 대응을 위한 이미지 대체 텍스트(Alt-text) 가이드를 마련한다.

색상만으로 정보를 구분하지 않고 아이콘/텍스트를 병기하는 원칙을 세운다.

고령자나 어린이 타겟일 경우 인터랙션 타겟 크기를 일반 대비 확대 설계한다.

사용자의 주요 사용 환경(실외 이동, 소음 환경 등)에 따른 알림 방식을 정의한다.

다국어 지원 시 국가별 날짜, 통화, 숫자 표기법(Locale) 정책을 정의한다.

모든 폼(Form) 요소에 레이블(Label)을 명시하여 접근성을 보장한다.

키보드만으로 서비스의 핵심 기능을 제어할 수 있는 접근성 순서를 정의한다.

사용자 그룹별(Guest, Member, Admin) 접근 가능한 메뉴 권한 지도를 작성한다.

회원가입 시 수집 정보 최소화 원칙을 준수하고 필수/선택 정보를 구분한다.

휴면 계정 전환 기준일과 안내 발송 시점을 비즈니스 정책으로 정한다.

장기 미사용자에 대한 혜택 소멸 및 계정 보호 정책을 수립한다.

서비스 이탈 방지를 위한 탈퇴 프로세스 내 '이탈 방지 제안'을 설계한다.

3. 기능 및 비즈니스 정책 (Functional Spec & Policy)
모든 기능 요구사항은 고유 ID(REQ-001 등)를 부여하여 추적 관리한다.

요구사항은 MoSCoW(Must, Should, Could, Won't) 우선순위를 지정한다.

기능 정의는 [누가]가 [무엇을 위해] [기능]을 사용하는지 사용자 스토리로 서술한다.

모든 비즈니스 정책에는 반드시 '정량적 수치'를 포함한다. (예: 48시간 유효)

프로세스별 승인/거절 단계의 판정 주체와 기준 수치를 명시한다.

포인트, 마일리지 등 재화의 발생, 사용, 소멸 정책을 수치화하여 정의한다.

쿠폰 발행 시 최소 주문 금액 및 최대 할인 한도를 명확히 제한한다.

검색 엔진의 자동 완성 로직과 검색 결과 정렬 가중치 정책을 수립한다.

추천 알고리즘 적용 시 추천 사유(사용자 기반, 아이템 기반 등)를 명시한다.

예약/결제 취소 시 시점별 환불 위약금 요율을 정책화한다.

동시성 이슈가 우려되는 액션(선착순 등)의 처리 우선순위 원칙을 정의한다.

사용자 간 거래 시 분쟁 조정을 위한 신고 및 제재 기준을 수치화한다.

대량 데이터 노출 시 페이징(Pagination) 방식과 기본 정렬 옵션을 정한다.

복수 선택 가능 항목의 경우 최대 선택 개수(Limit)를 기획 정책으로 정한다.

특정 등급 이상만 사용 가능한 기능의 진입 제한 문구를 정의한다.

업로드 가능한 파일의 확장자와 파일당 최대 용량 제한 수치를 명시한다.

데이터 입력창의 글자 수 제한(Min/Max) 수치를 기획 관점에서 정의한다.

자동 결제 서비스의 경우 결제 실패 시 재시도 주기와 횟수를 정의한다.

서비스 내 광고 노출 위치와 노출 빈도 제한(Frequency Cap) 정책을 세운다.

정산 데이터의 소수점 처리 방식(절사, 반올림)을 비즈니스 원칙으로 확정한다.

4. 사용자 경험 및 인터페이스 논리 (UX Logic)
사용할 디자인 시스템(ui-v3)과 컴포넌트 활용 원칙을 명시한다.

핵심 사용자 여정(Happy Path)을 시각적인 Flow 차트로 기술한다.

데이터 로딩 시 Skeleton UI 혹은 스피너 노출 정책을 상황별로 구분한다.

검색 결과가 없거나 데이터가 없는 Empty State의 안내 문구와 유도 액션을 정의한다.

작업 성공(Success) 시 사용자에게 주는 시각적 피드백과 다음 이동 경로를 정의한다.

작업 실패(Error) 시 노출할 오류 메시지의 톤앤매너와 해결 방안을 명시한다.

파괴적 액션(삭제 등) 수행 전 반드시 사용자 확인(Confirmation) 단계를 둔다.

뒤로 가기(Back) 버튼 클릭 시의 세션 유지 혹은 임시 저장 정책을 정의한다.

폼 입력 시 오타 방지를 위한 마스크(Mask) 처리 및 실시간 유효성 검사 정책을 세운다.

모바일/PC 등 디바이스별 레이아웃 대응 시 정보 노출의 우선순위를 결정한다.

중요 공지사항이나 팝업의 노출 기간 및 '오늘 그만 보기' 정책을 정의한다.

알림(Notification)의 유형별(Push, 알림톡, 배지) 발송 시점과 문구를 정의한다.

사용자의 첫 경험을 돕는 온보딩(On-boarding) 가이드 노출 로직을 설계한다.

하단 탭바나 메뉴의 메뉴 명칭과 아이콘 일관성 원칙을 수립한다.

스크롤 위치에 따른 Floating 버튼의 노출 및 숨김 로직을 정의한다.

이미지 슬라이더(Carousel)의 자동 재생 여부와 전환 간격 수치를 정한다.

긴 텍스트 노출 시 말줄임(Ellipsis) 처리 기준과 전체보기 제공 여부를 정의한다.

복잡한 기능 설명이 필요한 구간에 툴팁(Tooltip) 노출 정책을 수립한다.

다단계 프로세스(Wizard)의 경우 현재 단계와 전체 단계를 시각화한다.

인터랙션 반응 속도에 대한 사용자 체감 기준을 시각 피드백 정책에 반영한다.

5. 정보 모델 및 데이터 정책 (Information Model)
서비스 운영에 필요한 개념적 엔티티(핵심 정보 항목) 리스트를 작성한다.

정보 항목 간의 비즈니스적 관계(1:N 등)를 기획 관점에서 서술한다. (DB 설계 제외)

데이터의 **생명주기(Life-cycle)**를 정의한다: 생성, 보관, 삭제 시점.

수집하는 개인정보(PII) 항목을 식별하고 수집 근거 법령을 검토한다.

개인정보의 제3자 제공 시 제공 받는 자와 목적, 항목을 명시한다.

사용자가 본인의 데이터를 직접 수정/삭제할 수 있는 권한 범위를 정한다.

데이터 파기 시 즉시 삭제할지, 복구 기간(Soft Delete)을 둘지 결정한다.

개인정보 노출 화면의 마스킹(Masking) 처리 규칙을 정의한다. (예: 성명 * 처리)

서비스 성과 측정을 위해 수집해야 할 이벤트 로그(Event Log)를 정의한다.

업로드 이미지의 썸네일 생성 여부와 대표 이미지 선정 로직을 정의한다.

정산/금융 데이터의 경우 이력 추적(Audit Trail)을 위한 보관 정책을 세운다.

외부 API 연동 데이터의 갱신 주기(실시간/배치)를 비즈니스 관점에서 결정한다.

사용자 생성 콘텐츠(UGC)의 저작권 귀속 및 이용 허락 범위를 정의한다.

민감 데이터 저장 시 암호화 대상 항목을 비즈니스 요구사항으로 식별한다.

데이터 다운로드(Excel 등) 기능 제공 시 권한 범위와 보안 정책을 수립한다.

6. 예외 상황 및 오류 대응 (Exceptions)
결제 중 이탈 등 비즈니스 프로세스 중단 시의 자동 복구 정책을 수립한다.

서비스 점검 시 노출할 안내 페이지의 구성 요소와 긴급 공지 채널을 정의한다.

오프라인 상황(Network Offline)에서 작동 가능한 기능 범위를 명시한다.

기기 권한(GPS, 카메라 등) 거부 시 사용자에게 제시할 대체 액션을 정의한다.

잘못된 URL 접근 시 404 페이지 대응 및 메인 이동 링크를 제공한다.

검색 결과가 없을 때의 '연관 검색어' 혹은 '추천 상품' 노출 정책을 정한다.

중복 로그인 발생 시 세션 유지 정책(기존 로그아웃 등)을 결정한다.

잘못된 형식의 데이터 입력 시 필드별 에러 메시지 톤앤매너를 통일한다.

시스템 오류로 인한 데이터 유실 시 사용자 보상 가이드라인을 수립한다.

파일 업로드 실패 원인별(용량, 확장자, 개수) 메시지를 구분 정의한다.

특정 국가/IP 차단이 필요한 경우의 차단 안내 및 우회 방지 정책을 세운다.

타임아웃 발생 시 재시도 버튼 노출 여부와 안내 문구를 정의한다.

본인 인증 실패 시의 재시도 횟수 제한 및 잠금 해제 절차를 정의한다.

서비스 탈퇴 직후 동일 정보로 재가입 시의 제한 기간을 수치로 정의한다.

시스템 장애 시 기술 지원팀으로의 자동 리포팅 연동 여부를 결정한다.

7. 비기능 및 법적 준수 (Business Integrity)
사용자가 체감해야 할 최대 응답 시간(Latency) 목표를 비즈니스 관점에서 설정한다.

동시 접속자 수 예상치와 프로모션 시의 대기열 시스템 도입 여부를 정한다.

서비스 도메인별 필수 법령(위치정보법, 전자상거래법 등) 준수 여부를 확인한다.

글로벌 서비스의 경우 해당 국가의 데이터 주권 및 개인정보보호법(GDPR 등)을 검토한다.

장애 발생 시 사용자 고지 기한 및 보상 절차에 대한 CS 매뉴얼을 수립한다.

서비스 가용성(Uptime) 목표치를 비즈니스 운영 관점에서 명시한다.

보안 사고 발생 시의 비즈니스 대응 조직과 연락망을 정의한다.

저작권 침해 신고 발생 시의 콘텐츠 차단 및 소명 절차를 수립한다.

서비스 종료(Sunset) 시 데이터 반환 및 사전 고지 기간을 정의한다.

정산 정합성 검증을 위한 주기적인 데이터 대조(Audit) 정책을 수립한다.

8. 용어 및 인수인계 (Glossary & Handover)
기획-개발-QA 간 혼선을 막기 위한 서비스 전용 **용어 사전(Glossary)**을 작성한다.

기능별 완료 기준인 **인수 조건(Acceptance Criteria)**을 명확히 정의한다.

QA 단계에서 반드시 확인해야 할 비즈니스 크리티컬 시나리오를 선별한다.

향후 확장(Phase 2)을 고려한 현재 설계의 의도와 제약 사항을 기록한다.

프로젝트 종료 시 인도해야 할 최종 산출물(PRD, 가이드 등) 목록을 확정한다.

디자인 가이드 준수 여부를 검수하는 디자인 QA 체크리스트를 포함한다.

어드민(Admin) 페이지에서 관리해야 할 핵심 데이터 항목 리스트를 정의한다.

사용자 피드백(VOC) 수집 채널과 이를 서비스에 반영하는 프로세스를 수립한다.

페이지별 예외 규칙(Specific Exceptions)을 별도 섹션으로 구성한다.

개발 완료 후 잔여 작업(Backlog)의 우선순위를 비즈니스 가치 기반으로 재조정한다.
